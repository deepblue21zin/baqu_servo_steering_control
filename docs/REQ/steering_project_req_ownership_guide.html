1. 버그 6번 “닫혀있지 않다”의 정확한 의미

여기서 “닫혀있다”는 말은, 안전 관련 이벤트가 생겼을 때 감지 → 상태전이 → 출력 차단 → fault latch → 재기동 차단 → reset 조건 확인 → 복귀 가 하나의 정책으로 끝까지 연결돼 있다는 뜻이다.

현재 코드의 핵심 문제: 정지 동작은 일부 들어가 있지만, 그것을 누가 소유하는지, fault를 누가 붙잡고(latch) 있는지, 다시 Enable을 누가 허용하는지가 한 군데로 모여 있지 않다.

현재 코드에서 실제로 벌어지는 일

경로 현재 동작 왜 “안 닫힘”인가
position_control.c:405-418 safety evaluator가 unsafe를 반환하면 PositionControl_EmergencyStop()를 호출한다. fault를 “누적/latched” 상태로 관리하는 독립 state machine이 없다.
position_control.c:666-679 EmergencyStop은 PulseControl_Stop()와 Relay_Emergency()를 호출한다. 여기에는 Relay_ServoOff()가 없고, fault latch도 없다. 즉 “완전한 차단 정책”이 아니다.
position_control.c:558-565 PositionControl_Enable()는 별도 확인 없이 Relay_EmergencyRelease()를 수행한다. fault 원인이 아직 남아 있어도 다른 호출자가 쉽게 다시 enable 시킬 수 있다.
app_runtime.c:364-383 키보드 E로 enable, X로 estop 가능하다. 안전 복구 조건 확인 없이 사용자가 바로 재기동할 수 있다.
ethernet_communication.c:245-263 서비스 명령이 Relay_ServoOn(), Relay_ServoOff()를 직접 호출한다. actuator power 제어가 safety state machine을 우회할 수 있다.

이걸 한 문장으로 정의하면

현재 프로젝트는 “정지 동작”은 있지만,
“안전 상태를 끝까지 책임지는 단일 소유자(single owner)”가 없다.

이상적인 형태

[fault detect]
    ↓
[steering_safety_manager: SAFE 또는 FAULT 전이]
    ↓
[PulseControl_Stop]
[Relay_ServoOff]
[Relay_Emergency]
[fault_latched = 1]
[motor_enable_inhibit = 1]
    ↓
[operator reset + sensor valid + cmd alive + servo ready 확인]
    ↓
[READY 복귀]
중요: 지금 네가 keyboard 모드로 bring-up 중인 건 문제 아니다. 문제는 이 상태가 임시 bring-up 프로파일인지, 아니면 기본 동작으로 굳어버린 것인지가 코드에 드러나지 않는다는 점이다.

2. 현재 속도 병목과 수치 계산

현재 속도가 느린 이유는 하나가 아니라 3단 병목이다.

병목 단계 현재 값 코드 위치 의미 조향축 속도 환산
PID 출력 제한 output_limit = 10000 position_control.c:20-26 요청 주파수가 최대 10 kHz 수준으로 묶임 약 2.4 deg/s
Pulse clamp PULSECONTROL_MAX_FREQ_HZ = 100000 pulse_control.c:11-13 PulseControl이 100 kHz를 넘지 않도록 clamp 약 24 deg/s
TIM1 하드 ceiling PSC = 215, 최소 period 2 count tim.c:46-50, pulse_control.c:127-146 현재 prescaler로는 이론상 최대 약 416.7 kHz 약 100 deg/s
계산식은 아래와 같다.
steering_deg_per_s = pulse_hz × 0.003 motor_deg/pulse ÷ 12.5
여기서 0.003은 constants.h:17, 기어비 12.5는 constants.h:12.

이 말이 의미하는 것

  • 지금 당장 가장 낮은 병목은 PID 출력 제한 10k다.
  • 그 다음은 PulseControl clamp 100k다.
  • 나중에 100k 이상을 쓰고 싶으면 TIM1 prescaler 자체를 바꿔야 한다.
  • 따라서 “속도를 올린다”는 말은 단순히 Kp만 올리는 게 아니라, 주파수 경로 전체를 보는 일이다.

4. 엔코더 안정화 후 해야 할 수정

4-1. 지금은 왜 미루는가

  • 현재 가장 큰 병목은 `AO/BO -> level shifter -> TIM2` 외부 피드백 체인이다.
  • 24V 릴레이 연결과 startup gate를 지금 바꾸면, “안 돈다”의 원인이 제어정책인지 하드웨어인지 더 헷갈릴 수 있다.
  • bench 단계에서는 먼저 모터가 어떻게 도는지, 엔코더가 어떻게 읽히는지, 릴레이가 하드웨어에서 안정적으로 동작하는지를 닫는 게 우선이다.
  • 그다음에야 SAFE/FAULT, watchdog deadline, startup contract를 넣어도 “수정 때문에 테스트가 귀찮아진다”는 문제가 줄어든다.

4-2. 분리는 이렇게 진행하는 게 이해하기 쉽다

분리는 한 번에 크게 뜯는 게 아니라, 판단과 실행, 그리고 bench 경로와 제품형 경로를 분리하는 방식으로 진행하는 게 가장 안전하다.

단계 무엇을 분리하나 지금 코드에서 어디를 건드리나 왜 이 순서가 쉬운가
Step 1 진단과 제어 분리 `position_control.c` → `position_control_diag.c` 제어 동작은 유지하고 출력/문자열/상태출력만 밖으로 뺄 수 있다.
Step 2 안전 판단과 실제 차단 분리 `position_control_safety.c` + `PositionControl_EmergencyStop()` “unsafe 판단”은 밖으로 빼고, `PulseControl_Stop()` / `Relay_Emergency()` 같은 즉시 차단은 코어에 남길 수 있다.
Step 3 bench startup과 production startup 분리 `app_runtime.c` 지금의 auto-enable 경로를 `BENCH profile`로 남기고, 제품형 startup contract를 별도 분기에서 설계할 수 있다.
Step 4 top-level 상태기계 분리 `steering_safety_manager.*` 신규 `READY/ACTIVE/SAFE/FAULT`를 단일 owner에서 관리하게 되면 설명과 테스트가 쉬워진다.
Step 5 입력 freshness / watchdog / reset policy 연결 `app_runtime.c`, `ethernet_communication.c`, `iwdg.c`, `relay_control.c` sensor truth와 relay truth가 닫힌 뒤에 넣어야, 보호정지와 고장정지를 구분해서 검증할 수 있다. 
쉽게 이해하면:
1) 먼저 "보는 코드"와 "움직이는 코드"를 나눈다.
2) 그다음 "이상 판단"과 "즉시 차단"을 나눈다.
3) 마지막에 startup / SAFE / FAULT / reset 정책을 하나의 manager로 묶는다.

즉, 지금은 encoder feedback과 24V relay 테스트를 끝내고,
그 다음에 상태기계와 startup contract를 올리는 순서가 맞다.

5-2A. 목표 상태전이 명세 (Target Operating State Machine)

아래 표는 현재 코드가 이미 이렇게 동작한다는 뜻이 아니라, encoder feedback과 24V 릴레이 bench가 닫힌 뒤 steering controller를 어떤 운영 상태기계로 정리할지에 대한 목표 REQ다. 여기서 핵심은 SAFE는 보호정지, FAULT는 고장정지로 분리하는 것이다.

현재 상태 조건 다음 상태 출력 정책
INIT homing_done=1, angle_valid=1, estop_hw=0, servo_ready=1 READY motor_enable=0
READY cmd_alive=1, watchdog_ok=1, servo_alarm=0 ACTIVE motor_enable=1
ACTIVE cmd_rx_age_ms > 100 SAFE torque_limit=0, motor_enable=0, emg_assert=1
ACTIVE angle_valid=0가 50 ms 지속 FAULT motor_enable=0, emg_assert=1, fault_latched=1
ACTIVE overcurrent=1가 20 ms 지속 FAULT 또는 SAFE 즉시 torque limit 후 정지, 정책상 fault 여부 결정
ANY estop_hw=1 SAFE 즉시 정지, operator stop 정책
ANY servo_alarm=1 FAULT latched fault, 수동 reset 전 재기동 금지
SAFE / FAULT 수동 reset + 모든 입력 정상 READY 재시작 허용
SAFE와 FAULT의 구분 기준
SAFE: 통신 끊김, watchdog 이상, operator stop처럼 보호 목적으로 멈추는 경우
FAULT: sensor invalid, servo alarm, persistent overcurrent처럼 결함성 이벤트로 멈추는 경우
분류 대표 원인 복구 방식 왜 분리해야 하나
SAFE 명령 미수신, operator stop, watchdog freshness 실패 원인 해소 + reset policy 만족 시 READY 복귀 가능 외부 조건 회복으로 다시 움직일 수 있는 보호정지이기 때문
FAULT servo alarm, sensor invalid 지속, persistent overcurrent fault latch 유지, 수동 reset과 원인 제거가 필요 결함성 정지를 보호정지와 같은 방식으로 다루면 위험하기 때문
현재 코드와의 관계: 지금 소스는 아직 READY/ACTIVE/SAFE/FAULT top-level state owner를 갖고 있지 않다. 따라서 이 표는 현재 구현 설명이 아니라, encoder/relay bench가 끝난 뒤 steering_safety_manager에 반영할 목표 운영 명세로 읽어야 한다.

6. 내가 소유해야 하는 모듈과 방향성

  • 입출력과 실패 모드가 명확해서 테스트 케이스를 만들기 쉽다.
  • 지금은 미연결 상태로 먼저 만들어도 돼서, bring-up을 방해하지 않는다.

7. 꼭 알아야 하는 용어 정리

용어 뜻 이 프로젝트에서 왜 중요한가
Fail-safe 이상이 생기면 위험을 줄이는 방향으로 시스템이 전이되는 것 조향은 “계속 움직임”보다 “안전하게 멈춤”이 우선
Safe state 허용 가능한 최소 위험 상태 보통 pulse off, servo off, EMG assert 같은 출력으로 표현됨
Fault latch 한 번 발생한 fault를 붙잡아 두는 것 원인 제거 전 자동 재기동을 막기 위해 필요
Watchdog 소프트웨어가 살아 있는지 감시하는 장치 단순히 “있다”가 아니라 “몇 ms 내 safe인가”가 중요
Heartbeat / cmd_alive 상위 제어 명령이 살아 있음을 나타내는 신호 통신 끊김을 감지하는 핵심 입력
Pulse/Dir 펄스 수로 이동량, 방향 신호로 회전 방향을 주는 방식 지금 구동 경로의 기본 인터페이스
Quadrature encoder A/B 두 상의 위상차로 회전량과 방향을 읽는 엔코더 TIM2 encoder mode의 기반
4체배 엔코더 한 주기에서 4개의 에지를 모두 카운트하는 방식 해상도와 노이즈 민감도에 직접 영향
Settling time 목표 근처 허용 오차 범위 안으로 들어와 유지되기까지 시간 “빠르다”를 숫자로 말하는 핵심 KPI
Overshoot 목표를 넘어서 지나간 비율 또는 양 조향 응답 품질을 설명할 때 필수
Saturation 제어 출력이 상한/하한에 걸리는 것 지금 느린 속도의 직접 원인 중 하나
Plausibility 두 센서/두 경로가 서로 말이 되는지 검사하는 것 encoder와 pot을 같이 쓰는 이유
Bring-up 하드웨어와 펌웨어를 처음 살아나게 하고 확인하는 단계 지금 네가 하고 있는 테스트 단계

8. 꼭 알아야 하는 개념 정리

1) 단위 축 구분

motor deg와 steering deg를 절대 섞지 말아야 한다. 지금 코드에서도 기어비 12.5를 경유해 환산한다.

2) 관측과 제어 분리

현재 단계에서는 필터를 넣기보다 로그를 먼저 늘려야 원인 분리가 된다.

3) 상태와 출력 분리

safety manager는 판단만 하고, relay/pulse는 바인딩 계층이 실행하는 형태가 테스트에 유리하다.

4) 제어기 튜닝 ≠ 속도 상향

Kp를 올린다고 끝나지 않는다. output_limit, pulse clamp, timer ceiling이 같이 맞아야 한다.

5) SAFE와 FAULT의 차이

SAFE는 보호정지, FAULT는 고장정지로 보는 것이 설명이 쉽다.

6) Bring-up profile

기능을 끈 것은 괜찮다. 단, BENCH 전용이라는 표시가 반드시 남아야 한다.

현재 코드에서 구조상 보이는 특징

  • system_state.h의 상태 enum은 현재 업로드된 코드 기준으로 실제 소유자가 아니다. 즉, 상태 정의와 상태 운용이 분리돼 있다.

  • Relay_ServoOn/Off가 서비스 계층에서 직접 불릴 수 있어 actuator power path가 중앙 안전 로직을 우회할 수 있다.

10. 면접/이력서로 바꾸는 문장

이 프로젝트를 그냥 “조향 제어를 했다”로 말하면 약하다. 행동 + 수치 + 임팩트로 바꿔야 한다.
  • STM32F429ZI 기반 조향 제어 시스템에서 Pulse/Dir–Encoder–PositionControl 경로의 속도 병목을 수치화하고, PID 출력 제한(10 kHz), pulse clamp(100 kHz), TIM1 timer ceiling(약 416.7 kHz)의 다단 제한 구조를 분석해 조향 성능 개선 방향을 도출했다.
  • 엔코더/모터 bring-up 단계에서 log-only 진단 카운터와 latency/주파수 telemetry를 설계하여, 엔코더 불안정 현상을 감각이 아닌 데이터로 분석 가능한 구조로 전환했다.
  • 조향 프로젝트에서 READY/ACTIVE/SAFE/FAULT 기반 safety manager 설계 방향과 fault-latch 정책을 정의하고, 통신 미수신, 센서 invalid, estop, servo alarm에 대한 보호정지 요구사항을 구조화했다.
