Member Brief

Homing + Relay 담당 상세 할당서

이 문서는 startup safety 담당자가 지금 코드에서 어디까지 되어 있고 무엇이 아직 비어 있는지 기준점을 명확히 정리한 문서다. 현재 `relay_control.c`와 `homing.c`는 각각 최소 기능은 있지만, 실제 시스템은 아직 startup state machine 없이 자동 servo on + 자동 control enable로 동작한다. 따라서 이 파트의 가장 큰 과제는 기능 추가보다 운영 순서를 안전 계약으로 바꾸는 것이다.

Owner: Startup / Homing / Relay 관련 상위 REQ: REQ-P0-003 연결 파트: Sensor / Pulse / Control 기준 코드: 2026-03-30

이번 주 바로 맡길 일

  1. `app_runtime.c`의 자동 `Relay_ServoOn()`과 자동 `PositionControl_Enable()`을 state machine으로 대체한다.
  2. homing을 startup path에 실제로 넣고 성공 전에는 `RUN` 진입을 막는다.
  3. `SVON`, `EMG`, `pulse allow`, `control enable` truth table을 한 장으로 고정한다.

왜 이 파트가 급한가

sensor와 pulse가 좋아져도, 부팅 직후 자동 enable이 남아 있으면 시스템 안전 설명이 약하다. 실제 운영에서는 "어떻게 움직이나"보다 "언제 절대 움직이면 안 되나"가 먼저다.

현재 코드의 좋은 점

relay active level은 코드에 이미 반영돼 있고, homing도 ADC absolute angle을 encoder offset으로 연결하는 최소 골격은 있다. 즉 전부 새로 만드는 게 아니라, 현재 skeleton을 startup contract로 통합하는 작업이다.

Current Baseline

현재 코드에서 이미 구현된 것

파일 현재 구현 이미 되는 점 아직 부족한 점
Core/Src/relay_control.c `Relay_Init`, `Relay_ServoOn`, `Relay_ServoOff`, `Relay_Emergency`, `Relay_EmergencyRelease`가 active-low GPIO wrapper로 구현돼 있다. 기본 relay 제어 자체는 단순하고 명확하다. `SVON OFF`, `EMG released` 초기 상태도 명시돼 있다. shadow state, readback, sequencing delay, truth table, fault reason이 없다.
Core/Src/homing.c `ADC_Pot_GetSample()`로 steering angle을 읽고 motor axis로 변환한 뒤 encoder offset을 적용한다. ADC absolute sensor와 encoder offset을 연결하는 최소 homing skeleton은 있다. ADC validity bit를 검사하지 않고 timeout / retry / validation / failure reason이 없다.
Core/Src/app_runtime.c 초기화 후 `Relay_ServoOn()`, `HAL_Delay(500)`, `EncoderReader_Reset()`, `PositionControl_SetTargetWithSource(0deg)`, `PositionControl_Enable()`를 자동 실행한다. bring-up용 자동 시작 흐름이라 bench 테스트는 빠르다. homing이 startup path에 전혀 들어가 있지 않고, ready/arm 상태도 없다.
Core/Src/position_control.c `PositionControl_Enable()`가 `Relay_EmergencyRelease()`를 바로 호출하고, `PositionControl_EmergencyStop()`는 `Relay_Emergency()`를 호출한다. control layer와 relay layer가 최소한 연결은 돼 있다. EMG release가 startup gatekeeper가 아니라 enable 함수 안에 흩어져 있어 중앙집중 safety contract가 약하다.

Gap Analysis

지금 상태가 왜 부족한가

부팅 직후 자동 enable이 남아 있다

startup 직후 자동 `ServoOn`과 자동 `Control Enable`은 bring-up에는 편하지만, homing 완료나 ready precondition 없이 pulse path를 열 수 있다는 뜻이다.

Homing이 실제 startup flow에 통합되지 않았다

`homing.c`는 존재하지만 `app_runtime.c`에서 호출되지 않는다. 즉 코드상 homing 모듈이 있어도 현재 시스템 운전 조건과는 분리되어 있다.

Relay truth table이 없다

`SVON`, `EMG`, `control enable`, `pulse allow`가 어떤 상태에서 어떻게 조합되는지 표가 없다. 그 결과 enable/disable/estop/recovery가 모듈별로 흩어진다.

ESTOP과 reset recovery가 latch 계약으로 닫히지 않았다

emergency stop 동작은 있지만, 명시적 clear 절차와 reset 이후 safe startup 재진입 정책이 아직 없다.

Detailed REQ

팀원에게 그대로 넘길 실행형 요구사항

P0 REQ-HR-001

Startup state machine을 `app_runtime.c`에 실제로 넣는다

현재 구현: init 후 순차 호출로 자동 `ServoOn`과 자동 `Enable`이 실행된다.

왜 부족한가: boot, homing, ready, run, estop, recovery가 상태로 분리되지 않아 운영 가능 조건을 설명하기 어렵다.

개선 방향: 최소 `BOOT -> SERVO_OFF -> HOMING -> READY -> ARMED -> RUN -> ESTOP_LATCH -> RECOVERY` state를 정의하고, 각 상태에 entry action / timeout / failure action을 둔다.

완료 기준: 부팅 로그만 봐도 현재 state와 전이 이유를 알 수 있고, 자동 run 진입이 사라진다.

P0 REQ-HR-002

Homing을 startup path에 통합하고 validity check를 강제한다

현재 구현: `Homing_FindZero()`는 ADC sample 하나로 encoder offset을 적용하고 끝난다.

왜 부족한가: `validity` bit를 검사하지 않고, 실패 시 이유도 남기지 않으며, homing 성공 전에도 enable이 가능하다.

개선 방향: `ADC init -> sample validity -> reference decision -> encoder reset/offset -> homing validation` 순서를 startup state에 넣고, 실패 시 `READY`로 가지 않게 한다.

완료 기준: homing 성공 전 `RUN` 진입이 차단되고, failure reason이 로그와 상태 구조체에 남는다.

P0 REQ-HR-003

Relay truth table과 sequencing timing을 고정한다

현재 구현: relay 함수는 있지만 상태별 조합표와 timing contract가 없다.

왜 부족한가: `SVON`, `EMG`, pulse inhibit, control enable이 어느 상태에서 허용되는지 모르면 안전 리뷰가 어렵다.

개선 방향: state별 `SVON`, `EMG`, `Pulse allowed`, `Control enabled` 표를 만들고, `ServoOn` 이후 wait time과 `EMG release` 순서를 문서/코드에 맞춘다.

완료 기준: truth table과 GPIO wave capture가 서로 모순되지 않는다.

P1 REQ-HR-004

ESTOP을 latch 상태로 만들고 clear 절차를 분리한다

현재 구현: `PositionControl_EmergencyStop()`는 relay emergency를 호출하지만, clear policy는 별도 state machine으로 닫혀 있지 않다.

왜 부족한가: fault가 잠깐 풀렸다고 바로 다시 enable되면 안전성이 약하다.

개선 방향: `ESTOP_LATCH`와 명시적 clear API를 두고, `AUTO/MANUAL` 전환만으로 자동 복귀되지 않게 한다.

완료 기준: clear 절차 없이 `RUN`으로 돌아가지 않고, clear 시 precondition 체크 로그가 남는다.

P1 REQ-HR-005

READY precondition bitmask를 만든다

현재 구현: ready라는 중앙 상태가 없고, enable 함수가 개별 동작을 직접 열어 버린다.

왜 부족한가: 어느 조건이 부족해서 아직 운전 불가인지 파트 간 공유가 안 된다.

개선 방향: `homing_complete`, `sensor_valid`, `driver_ready`, `emg_released`, `fault_clear`, `manual_arm`을 bitmask로 관리하고 B 파트가 gatekeeper가 된다.

완료 기준: `READY=false` 이유를 한 번에 읽을 수 있고, 어떤 조건 하나라도 빠지면 run 진입이 차단된다.

P2 REQ-HR-006

Startup / recovery / reset event log와 negative test를 남긴다

현재 구현: startup state 전용 event log가 없고 reset cause 관리도 약하다.

왜 부족한가: 현장에서 왜 run을 막았는지, reset 이후 왜 safe state로 돌아갔는지 복원하기 어렵다.

개선 방향: state transition마다 `timestamp`, `from`, `to`, `reason`, `precondition result`를 남기고, watchdog / brownout / software reset 후 항상 safe startup으로 복귀하도록 한다.

완료 기준: cold boot, homing fail, estop, recovery, watchdog reset negative case 로그 세트가 존재한다.

Evidence

이 파트의 완료 증거로 받아야 하는 것

상태 전이

  • startup state diagram 1장
  • boot -> homing -> ready -> arm -> run 로그
  • estop -> clear -> recovery 로그

GPIO / timing

  • `SVON`, `EMG`, pulse inhibit sequencing 캡처
  • relay truth table
  • ServoOn wait time 근거

Negative tests

  • ADC invalid 상태 부팅
  • homing timeout
  • watchdog reset 후 safe startup 재진입
이 파트의 목표는 "relay가 동작한다"가 아니라 시스템이 언제 켜지고 언제 멈추고 언제 다시 올라와도 되는지를 코드 상태로 설명 가능하게 만드는 것이다.
