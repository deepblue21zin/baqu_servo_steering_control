Member Brief

Position Control 담당 상세 할당서

이 문서는 `position_control.c`를 맡는 팀원이 지금 코드에서 무엇이 이미 구현되어 있고, 무엇이 아직 현업형 제어기 계약으로 닫히지 않았는지 현재 baseline 기준으로 정리한 문서다. 현재 제어 루프 자체는 살아 있고 command lifecycle, PID, statistics, safety evaluator 분리까지는 되어 있다. 하지만 startup arm gate, fault latch, target validation, deadband/hold, richer diagnostics가 아직 부족하다.

Owner: Motion Control & Fault Manager 관련 상위 REQ: REQ-P0-004 기준 코드: 2026-03-30

이번 주 바로 맡길 일

  1. `PositionControl_Enable()`에 ready / arm precondition을 넣는다.
  2. 단일 `fault_flag`를 latch 가능한 reason 구조로 확장한다.
  3. target validation과 deadband/hold 정책을 controller 안에서 명시한다.

현재 코드의 강점

`position_control.c`는 PID, command lifecycle, settled 판정, stats, safety evaluator 분리까지 이미 되어 있다. 따라서 이 파트는 바닥부터 다시 만드는 일이 아니라, 이미 있는 루프를 production-like contract로 끌어올리는 작업에 가깝다.

가장 큰 리스크

지금은 startup에서 auto-enable이 남아 있고, `PositionControl_Enable()`도 ready 조건 없이 emergency release까지 수행한다. 즉 제어기는 동작하지만 운전 허가 정책은 아직 약하다.

Current Baseline

현재 코드에서 이미 구현된 것

파일 현재 구현 이미 되는 점 아직 부족한 점
Core/Src/position_control.c 1 ms 루프에서 `current_angle`, `error`, `velocity`, PID output을 계산하고 `PulseControl_SetFrequency()`로 보낸다. command lifecycle, reached/timeout/abort/fault 로그, stats, stable 판정이 이미 있다. `fault_flag`가 단일 값이고, `Enable()`이 ready precondition 없이 emergency release를 수행한다. 입력 validation과 rate limit도 없다.
Core/Src/position_control_safety.c angle limit, tracking error, velocity limit을 독립 evaluator로 판정한다. safety evaluator가 actuator side-effect 없이 result만 반환하는 구조는 좋다. stale sensor, wrong-direction, deadline miss, comm timeout, startup inhibit는 여기 taxonomy에 들어오지 않는다. 기본 timeout도 0이다.
Core/Src/position_control_diag.c command source/result/state 문자열과 debug vars를 갱신한다. `dbg_target_mdeg`, `dbg_pos_mdeg`, `dbg_err_mdeg`, `dbg_pwm_cmd`, `dbg_fault_flags`를 바로 볼 수 있다. actual dt, saturation, last fault time, ready reject reason, target source age 같은 분석 정보는 아직 부족하다.
Core/Src/app_runtime.c keyboard와 UDP 경로에서 steering target을 motor deg로 바꿔 `PositionControl_SetTargetWithSource()`로 넘긴다. 입력 source 구분과 CSV evidence는 이미 있다. startup auto-enable이 controller contract보다 앞에 있어, control ownership이 완전히 닫히지 않았다.

Gap Analysis

지금 상태가 왜 부족한가

운전 허가 조건이 제어기 안에서 닫혀 있지 않다

`PositionControl_Enable()`는 ready 검증 없이 `control_enabled = true`와 `Relay_EmergencyRelease()`를 수행한다. 이 상태에서는 startup safety와 control contract가 분리되지 않는다.

Fault manager가 너무 얇다

지금은 `fault_flag` 단일 값과 일부 debug bit만 있다. fault reason, latched 여부, clear 조건, safe action을 구분하기에는 부족하다.

입력 validation이 부족하다

`SetTargetWithSource()`는 범위만 검사하고 `NaN`, `Inf`, stale source, 급격한 step, rate limit을 처리하지 않는다. 특히 `NaN`은 비교 연산을 통과할 수 있어 명시적 `isfinite()` 검사가 필요하다.

정지 수렴 정책이 lifecycle과 완전히 일치하지 않는다

stable 판정은 있지만 deadband / hold / release zone이 별도 계약으로 정리돼 있지 않아, 작은 오차에서 pulse가 계속 살아 있을 가능성을 체계적으로 다루지 못한다.

Detailed REQ

팀원에게 그대로 넘길 실행형 요구사항

P0REQ-CTRL-001

Enable은 ready / arm 조건이 만족될 때만 허용한다

현재 구현: `PositionControl_Enable()`는 즉시 `control_enabled=true`와 `Relay_EmergencyRelease()`를 수행한다.

왜 부족한가: homing, relay, sensor validity, operator arm이 아직 제어기 gate로 들어오지 않는다.

개선 방향: `ready`, `fault_clear`, `homing_complete`, `emg_released` 같은 precondition bitmask를 받아 enable 거부 사유까지 남긴다.

완료 기준: power-on 직후 아무 입력이 없을 때 pulse가 없고, `READY=false`면 enable 거부 로그가 남는다.

P0REQ-CTRL-002

Fault manager를 bitmask + latch + clear matrix로 확장한다

현재 구현: timeout / limit / velocity는 일부 구분되지만 내부 fault representation은 매우 얇다.

왜 부족한가: 같은 emergency stop이어도 tracking, timeout, stale sensor, estop input은 후속 조치가 다르다.

개선 방향: reason bitmask, latched 여부, clear 조건, safe action을 표로 정하고 `PositionControl_Enable()`에서 자동 clear를 금지한다.

완료 기준: fault 발생 후 reason을 snapshot과 구조체에서 읽을 수 있고, clear 조건 전에는 재enable이 거부된다.

P1REQ-CTRL-003

Target input validation과 rate limit를 추가한다

현재 구현: motor_deg 범위 비교만 수행한다.

왜 부족한가: `NaN`, `Inf`, stale source, 너무 큰 step가 제어기를 직접 흔들 수 있다.

개선 방향: `isfinite`, range, source freshness, optional slew/rate limit를 추가하고 source별 reject reason을 로그로 남긴다.

완료 기준: invalid target은 명확히 거부되고, 큰 step에서도 output이 제한된 기울기로 변한다.

P1REQ-CTRL-004

Deadband / hold / release 정책을 정식화한다

현재 구현: `STABLE_ERROR_THRESHOLD`와 `STABLE_TIME_MS` 기반으로 reached를 판정한다.

왜 부족한가: stable 판정과 actual pulse stop 정책이 완전히 같은 의미로 설명되지는 않는다.

개선 방향: `|error|`와 `|velocity|`를 함께 보는 hold zone, release zone, hysteresis 정책을 넣고 steady-state hunting을 줄인다.

완료 기준: `0 -> 5deg -> 0deg` step 후 목표 근처에서 pulse가 멈추고 steady-state output이 0으로 수렴한다.

P1REQ-CTRL-005

Safety taxonomy를 stale / wrong-direction / comm timeout까지 확장한다

현재 구현: safety evaluator는 angle, tracking, velocity만 본다.

왜 부족한가: 실제 현장 문제는 sensor stale, commanded direction mismatch, comm timeout이 더 흔할 수 있다.

개선 방향: sensor team과 integration team 결과를 받아 stale/mismatch/timeouts를 controller fault taxonomy에 포함한다.

완료 기준: 각 fault가 detect condition, latch 여부, clear 절차와 함께 정의된다.

P2REQ-CTRL-006

제어기 진단과 성능 지표를 정량화한다

현재 구현: stats, CSV, debug vars는 있지만 tuning evidence 형식이 아직 정해져 있지 않다.

왜 부족한가: gain 변경 전후 비교와 fault 재현 설명이 체계적이지 않다.

개선 방향: actual dt, saturation, last fault timestamp, target source, settle/overshoot CSV 규격을 정하고 tuning result 표로 관리한다.

완료 기준: 대표 step response 3종에 대해 CSV와 요약표를 남기고 before/after 비교가 가능하다.

Evidence

이 파트의 완료 증거로 받아야 하는 것

로그

  • enable reject log
  • fault reason / clear log
  • step response CSV

문서

  • fault matrix
  • enable precondition 표
  • deadband / hold parameter 표

관찰 도구

  • CSV target/current/error/output
  • `dbg_target_mdeg`, `dbg_pos_mdeg`, `dbg_err_mdeg`, `dbg_pwm_cmd`
  • `PE9` stop 파형
이 파트의 최종 목표는 "PID가 돈다"가 아니라 언제 움직이고 언제 멈추고 왜 fault를 냈는지까지 설명 가능한 제어기를 만드는 것이다.
