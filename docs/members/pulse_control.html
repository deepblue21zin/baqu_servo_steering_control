Member Brief

Pulse Control 담당 상세 할당서

이 문서는 `pulse_control.c`를 맡는 팀원이 다음으로 무엇을 닫아야 하는지 현재 코드 사실 위에서 정리한 문서다. 지금은 PE9 pulse, PE10 direction, reverse guard, requested/applied status까지 구현돼 있어 actuator interface의 뼈대는 좋다. 하지만 ready gate, residual pulse 기준, step/continuous contract, hardware evidence가 아직 부족하다.

Owner: Pulse / Driver Timing 관련 상위 REQ: REQ-P0-002 관련 상위 REQ: REQ-P1-001 관련 상위 REQ: REQ-P2-001

이번 주 바로 맡길 일

  1. 현재 runtime contract를 `100000 Hz clamp + 1 ms reverse guard` 기준으로 문서/코드에 통일한다.
  2. ready / servo / emg / control enable과 pulse inhibit 조건을 연결한다.
  3. `PE9`, `PE10`, `[ENCDBG]`, drive monitor를 한 장면으로 묶는 검증 matrix를 만든다.

왜 이 파트가 중요하나

controller가 출력한 signed command가 실제 servo drive가 받아들일 수 있는 파형으로 바뀌는 구간이 바로 여기다. 센서나 제어기 문제가 아니라도 pulse contract가 흐리면 전체 시스템이 흔들린다.

현재 코드의 강점

TIM1 actual clock 계산, direction polarity macro, line-driver enable, reverse guard, requested/applied Hz status가 이미 있어 기본 actuator interface는 상당히 좋아졌다. 다음 단계는 기능 추가보다 운영 계약과 증거 체계를 닫는 일이다.

Current Baseline

현재 코드에서 이미 구현된 것

파일 현재 구현 이미 되는 점 아직 부족한 점
Core/Src/pulse_control.c `PE9(TIM1_CH1)` pulse, `PE10` direction, shared line-driver enable, 10~100000Hz clamp, 1ms reverse guard를 구현한다. continuous frequency path와 `requested_frequency_hz / applied_frequency_hz / reverse_guard_active` status가 이미 있다. system ready나 relay state와 직접 연결된 output gate가 없다. reverse / stop 후 residual pulse 허용 기준도 문서화돼 있지 않다.
Core/Inc/pulse_control.h `PulseControl_Status_t`가 `requested`, `applied`, `arr`, `compare`, `direction`, `output_active`, `busy`를 제공한다. 현재 출력 상태를 런타임에서 비교적 잘 재구성할 수 있다. `last_stop_reason`, `gate_reason`, `mode`, `last_reverse_reason` 같은 triage 정보가 없다.
Core/Src/app_runtime.c CSV와 keyboard snapshot에서 `REQ`, `AP`, `RUN`, `REV`, `DIR`을 같이 기록한다. bench에서 actuator 상태를 로그로 확인할 수 있다. scope / drive monitor / encoder evidence를 같은 실험 세트로 묶는 규칙이 아직 없다.
Core/Inc/constants.h `MAX_PULSE_FREQ`가 1000000으로 남아 있다. future design target을 남겨 둔 흔적으로는 볼 수 있다. 현재 runtime clamp 100000Hz와 분리 표기가 없어 현장 설명 시 혼선을 준다.

Gap Analysis

지금 상태가 왜 부족한가

Current contract가 하나로 정리돼 있지 않다

코드 한쪽은 100kHz clamp를 쓰고 다른 문맥에서는 1MHz 상수가 남아 있어, "지금 실제로 얼마까지 안전하게 쓸 수 있나"를 한 문장으로 답하기 어렵다.

Ready gate가 actuator까지 내려오지 않았다

startup, homing, relay, estop 정책이 pulse output inhibit 조건으로 아직 닫히지 않았다. 상위가 실수해도 여기서 한 번 더 막아줄 방어선이 필요하다.

Mode contract가 아직 얇다

continuous와 step mode가 같은 타이머를 쓰지만, 어떤 시점에 누가 ownership을 가지는지, step mode 주파수 기준이 무엇인지 명확하지 않다.

Waveform evidence가 시스템 증거로 묶여 있지 않다

requested / applied 숫자만으로는 충분하지 않다. `PE9`, `PE10`, drive monitor, real encoder 응답을 같은 기준으로 캡처해야 actuator closure라고 말할 수 있다.

Detailed REQ

팀원에게 그대로 넘길 실행형 요구사항

P0 REQ-PULSE-001

Current runtime pulse contract를 하나로 고정한다

현재 구현: runtime은 10~100000Hz clamp를 적용하지만 `constants.h`에는 `MAX_PULSE_FREQ = 1000000`이 남아 있다.

왜 부족한가: 현재값과 future target이 섞여 있으면 bench 결과 해석과 안전 범위 설명이 흐려진다.

개선 방향: current runtime contract와 future design target을 분리해 문서화하고, status / README / portal 설명도 같은 기준으로 맞춘다.

완료 기준: 코드, 문서, 포털 어디서 봐도 같은 최대 주파수와 guard time을 말한다.

P0 REQ-PULSE-002

Ready / Servo / EMG 조건을 output inhibit로 묶는다

현재 구현: `PulseControl_SetFrequency()`는 자체적으로 direction과 guard를 처리하지만 startup readiness를 직접 보지 않는다.

왜 부족한가: 상위 호출 실수나 startup race가 있으면 준비되지 않은 상태에서 pulse가 나갈 수 있다.

개선 방향: `system_ready`, `servo_on`, `emg_released`, `control_enabled`, `homing_complete` 같은 gate 입력을 정의하고, output stage에서 최종 inhibit를 건다.

완료 기준: `READY=false` 또는 `EMG active`일 때 어떤 frequency 요청도 실제 pulse로 나가지 않는다.

P0 REQ-PULSE-003

Reverse / stop timing contract와 residual pulse 기준을 정량화한다

현재 구현: `stop -> 1ms wait -> dir change -> 1ms wait -> restart` 구조의 reverse guard가 있다.

왜 부족한가: guard가 있다는 것과 실제 파형에서 glitch가 없는 것은 다르다.

개선 방향: reverse, disable, estop 시 residual pulse 허용량과 측정 방법을 정하고 logic analyzer 캡처로 검증한다.

완료 기준: 100회 reverse 반복에서 glitch count와 residual pulse가 허용 기준을 만족한다.

P1 REQ-PULSE-004

Status 구조체를 triage 가능한 수준으로 확장한다

현재 구현: status는 requested/applied/ARR/CCR/direction/busy를 제공한다.

왜 부족한가: 문제가 생겼을 때 왜 멈췄는지, 왜 gate 되었는지, step인지 continuous인지 구분하기 어렵다.

개선 방향: `last_stop_reason`, `last_reverse_reason`, `gate_reason`, `mode`, `line_driver_state`를 추가해 사후 분석 가능성을 높인다.

완료 기준: UART snapshot 하나로 actuator 상태 전이를 재구성할 수 있다.

P1 REQ-PULSE-005

Continuous mode와 step mode의 경계를 명확히 한다

현재 구현: `PulseControl_SendSteps()`가 존재하지만 별도 step frequency 입력이나 mode ownership 문서가 없다.

왜 부족한가: 같은 타이머를 두 모드가 공유할 때 stop/start 시 꼬임이 생기기 쉽다.

개선 방향: mode enum, busy semantics, transition rule, explicit step frequency 또는 기본값 정책을 정의한다.

완료 기준: mode transition negative test에서 continuous와 step이 동시에 타이머를 건드리지 않는다.

P2 REQ-PULSE-006

Actuator closure evidence pack을 만든다

현재 구현: 로그는 좋지만 scope, drive monitor, real encoder evidence가 같은 matrix로 정리돼 있지 않다.

왜 부족한가: 현재 bench 병목이 actuator path인지 sensor truth인지 빠르게 분리하려면 같은 기준의 증거 묶음이 필요하다.

개선 방향: `100Hz / 1kHz / 10kHz / stop / reverse / estop` 실험별로 `requested`, `applied`, `PE9`, `PE10`, `ENCDBG`, drive monitor를 같이 캡처한다.

완료 기준: wave matrix 1장과 대표 캡처 세트만으로 actuator path 상태를 설명할 수 있다.

Evidence

이 파트의 완료 증거로 받아야 하는 것

파형

  • `PE9` low / mid / high frequency 캡처
  • `PE9 + PE10` reverse guard 동시 캡처
  • disable / estop stop latency 캡처

로그

  • `requested_hz` vs `applied_hz` 비교 표
  • gate reason / stop reason snapshot
  • `ENCDBG`와 drive monitor 동시 비교 로그

문서

  • current runtime pulse contract 표
  • output inhibit truth table
  • mode transition contract
이 파트의 핵심은 "pulse가 나온다"가 아니라 언제 나가도 되고 언제 절대 나가면 안 되는지까지 포함한 actuator contract를 닫는 것이다.
